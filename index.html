01 · Core topology

System architecture

The platform starts as a modular monolith for delivery speed, but its boundaries already align with future event-driven service extraction.

Platform map

Operational flow

Clients
Web and mobile experiences for patients, operators, technicians, and administrators
Gateway
Authentication, rate control, tenancy awareness, and route entry
Services
Patients, orders, samples, results, billing, reporting, and operational rules
Data + AI
Postgres, Redis, object storage, event streams, and model-driven decisioning

Design principles

  • Modular monolith for MVP simplicity.
  • Service boundaries preserved for future extraction.
  • Async processing where lab flow benefits from eventual consistency.

Architecture stance

Keep synchronous APIs for operator-critical workflows, and introduce events only where lifecycle transitions, reporting, and integrations create real separation value.

02 · Interface contracts

API contracts

APIs are framed as stable product surfaces: explicit payloads, idempotent mutations, tenant isolation, and security as baseline behavior.

REST surface
Create order
POST /api/v1/orders Request: { "patient_id": "uuid", "tests": ["cbc"] } Response: { "order_id": "uuid", "status": "CREATED" }

Contract intent

A creation request should immediately return a canonical identifier and durable lifecycle entry state.

Submit result
POST /api/v1/results { "order_test_id": "uuid", "values": [ { "parameter": "HB", "value": 13.5 } ] }

Rules

  • Idempotency for POST operations.
  • JWT-based authentication and claims-driven access.
  • Strict tenant boundary enforcement at every resource edge.

Execution flow

  • Receive domain event.
  • Evaluate conditions and business rules.

09 · Delivery scope

MVP scope

The first release should optimize for operational usefulness, low complexity, and short feedback loops. Advanced architecture should stay visible but intentionally deferred.

Build now vs later
